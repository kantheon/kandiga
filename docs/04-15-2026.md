# Kandainer Voice Pipeline — Overnight Build Log (April 15-16, 2026)

**Goal:** Sub-0.91× RTF, under 5 GB total RAM, Qwen 3.5 brain + voice cloning. Works for ANY length text.

**Constraint:** PersonaPlex proven at 0.9× RTF with clean Hakim voice. Must find a way to connect Qwen's intelligence to PersonaPlex's voice WITHOUT breaking audio quality.

## Key Discovery: PersonaPlex Cannot Be Used As Forced-Text TTS

Exhaustively tested every text-injection approach:
| Method | Result |
|---|---|
| Forced text_token per frame | Audio degrades after ~35 words |
| Biased logits (bias=5-50) | Same degradation as forced |
| Self-paced guided (model controls timing) | 100% text tokens hit but audio says DIFFERENT words |
| Chunked with warm-state resets | Each chunk still garbles complex phrases |

**Root cause:** PersonaPlex's text and audio channels are loosely coupled. The depformer generates audio from the temporal transformer's hidden state, not from text tokens directly. External text injection (forced or biased) doesn't control what the audio actually sounds like. The model was trained for free-form full-duplex conversation, not text-to-speech.

## Approach: Persona-Relay Architecture

Instead of injecting text into PersonaPlex's generation loop, use the **persona/system prompt** as the bridge:

```
User speaks → PersonaPlex STT (text channel, free) → user text
                                                        ↓
                                                   Qwen 3.5 brain
                                                        ↓
                                                   response text
                                                        ↓
                                            PersonaPlex persona update:
                                            "Respond with: {response text}"
                                                        ↓
                                            PersonaPlex generates audio NATURALLY
                                            (no forced tokens, model speaks freely)
```

PersonaPlex sees Qwen's answer as CONTEXT in its system prompt, then generates audio in its own natural way. The text may not be verbatim but should be semantically faithful.

**Why this might work:** PersonaPlex reliably follows persona instructions (proven: "You are a receptionist" → always greets correctly). We're extending this to include specific answer content in the persona.

### Architecture (the solution)

```
Session start:
  Qwen 3.5 prepares comprehensive knowledge → PersonaPlex persona
  PersonaPlex warms up with rich persona (5s one-time, 20ms cached)

Each turn:
  User speaks → PersonaPlex processes + responds from persona knowledge
  No forced text, no injection — pure natural conversation
  
  For dynamic lookups (5% of interactions):
    Capture user question → Qwen 3.5 + tool call
    Inject short result as forced text (<15 tokens, within safe range)
    PersonaPlex continues naturally
```

### Building Production Pipeline

Combining all proven optimizations:
1. Rich persona (Qwen prepares knowledge upfront OR pre-configured)
2. Pipelined Mimi encode/decode (ThreadPoolExecutor, 2 Tokenizer instances)
3. 8-slice depformer (skip unused user channels)
4. Warm-state caching (20ms reload between sessions)
5. TurboQuantCache for long conversations (3-bit KV cache compression)
