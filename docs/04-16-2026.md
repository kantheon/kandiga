### 15:50 — Post-mortem complete. New approach: LoRA fine-tune the Talker (Option D)

**Post-mortem on adapter branch:**
The fundamental error was optimizing hidden-state similarity (MSE/cosine) when the actual target should have been codec-token prediction (cross-entropy). The Talker is extremely sensitive to its input distribution — even cosine 0.50 states (which sounds high) produce completely unrelated codec sequences. Representation spaces between different model families (Qwen2 vs Qwen3.5) cannot be bridged by projection alone.

**Four options evaluated:**
- ~~Option A: Qwen3-Omni-30B-A3B~~ — rejected, too slow even with SEM on 16GB Mac
- ~~Option B: Codec distillation~~ — rejected, user has spent too many hours on codec/SNAC training paths already
- ~~Option C: SNAC training v7~~ — rejected, same reason as B
- **Option D: LoRA fine-tune the Talker** — SELECTED

**Option D design:**
```
Qwen 3.5 4B (FROZEN) → 2560-dim hidden states
    → New projection (2560→896, TRAINED)
    → Talker + LoRA rank-16 (384M base + ~4M LoRA, TRAINED)
    → codec_head → codec tokens
    → Token2Wav (FROZEN) → audio
```

**Key difference from failed adapter approach:**
- Before: trained adapter against HIDDEN STATES (MSE/cosine) — wrong target
- Now: train against CODEC TOKENS (cross-entropy) — the actual output that produces speech
- The Talker gets to ADAPT its internal attention patterns to Qwen 3.5's representation style
- LoRA preserves pretrained codec generation knowledge while allowing adaptation

**Training plan:**
1. Generate training data: run Omni on 2000-5000 texts → capture (text, Qwen 3.5 hidden states, ground truth codec tokens)
2. Replace `thinker_to_talker_proj` with new `Linear(2560→896)`
3. Add LoRA (rank 16) to Talker Q/K/V/O attention projections
4. Train with cross-entropy on codec tokens
5. Trainable params: ~6M (2.3M proj + ~4M LoRA)
6. Production RAM: ~2.6 GB (Qwen 3.5 1.5GB + Talker 200MB + Token2Wav 900MB)

### 18:48 — PIVOT: Codec decoder on Qwen 3.5 (encoder-decoder). LoRA approach abandoned.

**Opus 4.7 audit** (user switched to 4.7 model for second opinion) identified the real issue:

1. Training LoRA on a 4-bit quantized Talker is a hack — gradients flow through quantization noise
2. Starting manifold is bad — zero AR agreement at init means training is digging signal out of noise, not refining a plausible starting point
3. The Omni Talker can stay frozen because **we don't need it**. The Omni Token2Wav is what we actually want (codec → audio). The Talker just maps text→codec — something Qwen 3.5 can do directly.

**New approach: codec decoder on Qwen 3.5 (encoder-decoder).**

```
Frozen Qwen 3.5 3-bit (encoder): text → text_hidden [B, T, 2560]
    ↓ (no gradient, mx.stop_gradient)
New codec decoder (trained from scratch):
    - Codec embeddings (8448 × 512)
    - 6 transformer layers, d=512, 8 heads, FF=2048
    - Self-attention + cross-attention to Qwen hidden states
    - Codec head (512 → 8448)
    ↓
Codec tokens → Omni Token2Wav (frozen) → audio
```

**Why this sidesteps all prior problems:**
- No gradients through quantized model (encoder-decoder split)
- Clean from-scratch training of a small decoder
- No hidden state matching (whole wrong paradigm)
- No fighting foreign representation spaces
- Standard encoder-decoder setup, well-understood

**Training started 18:48:**

- 2731 (text, codec_tokens) pairs
- 30 epochs, batch 8×4 grad accum (effective 32)

- Process alive (11% CPU, 12.1% MEM), epoch 1 in progress

**Why earlier attempt failed:**
- First try had MLX VJP error on CustomKernel — gradients can't propagate through quantized Qwen layers when codec embeddings are injected at input level

**Probability estimate (Opus 4.7):** 50-60% this works. It's standard VALL-E/Whisper-style architecture — proven to work with discrete codec vocabularies. The main risk is teacher-forcing/rollout gap, which AR eval will catch immediately.
